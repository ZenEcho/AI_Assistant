# 启动与打包

这份文档聚焦“怎么把项目跑起来、测起来、打出来”。如果你只关心架构，请先看 [技术栈与开发指南](./技术栈与开发指南.md)；如果你要发布或回归本轮新增能力，这份文档是第一入口。

## 1. 环境要求

基础环境：

- `Node.js 20+`
- `pnpm 9+`
- `Rust stable`（需确保 `cargo -V` 可执行）

Windows 下额外建议：

- `Visual Studio Build Tools 2022`
- `Desktop development with C++` / `Microsoft.VisualStudio.Workload.VCTools`
- `WebView2 Runtime`
- 如需生成 `MSI`，确认系统启用了 `VBSCRIPT` 可选功能

Windows 上如果使用普通 `PowerShell`，推荐通过仓库脚本启动：

```bash
pnpm tauri:dev:windows
```

这个命令会先导入 `C:\Program Files (x86)\Microsoft Visual Studio\2022\BuildTools\Common7\Tools\VsDevCmd.bat` 的环境变量，再执行 `pnpm tauri dev`。如果你本身就在 `Developer PowerShell for VS 2022` 中，则可以直接运行 `pnpm tauri dev`。

官方参考：

- [Tauri Prerequisites](https://v2.tauri.app/start/prerequisites/)
- [Tauri Distribute](https://v2.tauri.app/distribute/)

## 2. 安装依赖

```bash
corepack enable
pnpm install
```

## 3. 本地启动方式

### 3.1 仅启动前端

```bash
pnpm dev
```

适合：

- 调页面样式
- 调普通表单交互
- 调不依赖 Tauri 的纯前端逻辑

默认端口：`http://localhost:1420`

### 3.2 启动桌面应用

```bash
pnpm tauri dev
```

这是日常开发主入口。当前行为是：

1. Tauri 读取 `src-tauri/tauri.conf.json`
2. 先执行 `beforeDevCommand`
3. `beforeDevCommand` 调用 `scripts/tauri-before-dev.mjs`
4. 脚本检查 `1420` 端口是否已存在本项目的 Vite 服务
5. 如果存在则复用，不存在则自动拉起 `pnpm dev`
6. Vite 就绪后启动桌面壳

这意味着大多数情况下直接执行 `pnpm tauri dev` 就够了。

如果当前终端不是 `Developer PowerShell for VS 2022`，请优先执行：

```bash
pnpm tauri:dev:windows
```

### 3.3 使用 Smoke 配置启动

如需隔离主应用配置、验证 Smoke 包逻辑，可显式指定 Smoke 配置：

```bash
pnpm tauri dev --config src-tauri/tauri.smoke.conf.json
```

Smoke 配置与正式配置的主要区别：

- `productName` 为 `AI Assistant Smoke`
- `identifier` 为 `com.example.aiassistant.smoke`
- 会使用独立的应用数据目录，避免污染日常配置

## 4. 常用命令清单

| 命令 | 用途 |
| --- | --- |
| `pnpm dev` | 启动前端开发服务器 |
| `pnpm tauri dev` | 启动桌面开发模式 |
| `pnpm tauri:dev:windows` | 在普通 PowerShell 中注入 VS 开发环境后启动桌面开发模式 |
| `pnpm preview` | 预览前端构建结果 |
| `pnpm typecheck` | TypeScript 类型检查 |
| `pnpm test` | Vitest 交互模式 |
| `pnpm test:run` | 单次运行前端测试 |
| `pnpm test:native` | 运行 Rust 侧 `system_input` 相关测试 |
| `pnpm test:system-input` | 运行系统输入增强相关前端 + Rust 测试 |
| `pnpm build` | 构建前端 `dist/` |
| `pnpm tauri build` | 构建桌面应用 |
| `pnpm build:desktop` | 桌面构建 + 产物归档 |
| `pnpm build:desktop:smoke` | 生成 Smoke 版桌面构建 |
| `pnpm tauri:artifacts` | 对已有构建结果执行产物归档 |
| `pnpm smoke:system-input:notepad` | 基于默认安装/构建结果执行 Notepad Smoke |
| `pnpm smoke:system-input:notepad:smokeapp` | 基于 Smoke 包执行 Notepad Smoke |
| `pnpm storybook` | 启动 Storybook |
| `pnpm build-storybook` | 构建 Storybook |

## 5. 推荐开发回路

### 5.1 改普通前端页面

推荐顺序：

```bash
pnpm dev
pnpm typecheck
pnpm test:run
```

### 5.2 改翻译链路、自动目标语言、历史缓存

推荐顺序：

```bash
pnpm tauri:dev:windows
pnpm typecheck
pnpm test:run
```

如改动涉及自动目标语言，建议同时按 [自动目标语言测试用例](./自动目标语言测试用例.md) 做手工回归。

### 5.3 改系统输入增强

推荐顺序：

```bash
pnpm tauri:dev:windows
pnpm test:system-input
pnpm build:desktop:smoke
pnpm smoke:system-input:notepad:smokeapp
```

如果只是本地快速验证，也可以先直接执行：

```bash
pnpm smoke:system-input:notepad
```

## 6. Smoke 测试说明

Notepad Smoke 脚本位于：

- [`scripts/system-input-smoke-notepad.ps1`](../scripts/system-input-smoke-notepad.ps1)

它用于半自动验证：

- 应用是否能拉起
- 配置是否能写入
- 外部输入框是否能被触发
- 译文是否能回写或产生可观测结果

### 6.1 直接跑默认构建结果

```bash
pnpm smoke:system-input:notepad
```

脚本会优先尝试这些可执行文件：

- `artifacts/tauri/v0.1.0/windows-x64/portable/ai-assistant.exe`
- `src-tauri/target/release/ai-assistant.exe`
- `src-tauri/target/debug/ai-assistant.exe`

### 6.2 跑 Smoke 包

```bash
pnpm build:desktop:smoke
pnpm smoke:system-input:notepad:smokeapp
```

这个组合更适合发版前验证，因为它使用独立的 Smoke 配置路径：

- `%APPDATA%\\com.example.aiassistant.smoke\\app-config.json`

### 6.3 常用脚本参数

如果需要手工调试 PowerShell 脚本，可关注这些参数：

- `-AppExecutable`
- `-ConfigPath`
- `-TriggerMode`
- `-CaptureMode`
- `-WritebackMode`
- `-TargetLanguage`
- `-SkipAppRestart`
- `-KeepNotepad`

## 7. 构建与打包

### 7.1 仅构建前端

```bash
pnpm build
```

输出目录：

- `dist/`

### 7.2 构建桌面应用但不打安装包

```bash
pnpm tauri build -- --no-bundle
```

适合：

- 先确认 Rust 与前端都能编译通过
- 排查 Tauri / Rust 构建错误
- CI 中做快速构建校验

### 7.3 构建桌面安装包

```bash
pnpm tauri build
```

当前配置会自动：

1. 先执行 `pnpm build`
2. 读取 `dist/` 作为前端资源
3. 生成原生可执行文件
4. 输出平台支持的安装包格式

常见输出目录：

- `src-tauri/target/release/`
- `src-tauri/target/release/bundle/`

### 7.4 打包并归档交付产物

```bash
pnpm build:desktop
```

或在已经打包完成后单独执行：

```bash
pnpm tauri:artifacts
```

归档脚本位于：

- [`scripts/classify-tauri-artifacts.mjs`](../scripts/classify-tauri-artifacts.mjs)

归档后的目录形态：

```text
artifacts/
└─ tauri/
   └─ v0.1.0/
      └─ windows-x64/
         ├─ installers/
         │  ├─ msi/
         │  └─ nsis/
         ├─ portable/
         ├─ symbols/
         └─ summary.json
```

语义说明：

- `installers/`：给用户安装的包
- `portable/`：直接运行的便携版程序
- `symbols/`：调试符号
- `summary.json`：归档清单与来源说明

以后如果只是找“真正要上传 / 分发的文件”，优先看 `artifacts/tauri/...`，不要直接在 `target/release` 里人工挑。

## 8. 发布前最小检查清单

建议至少执行：

1. `pnpm typecheck`
2. `pnpm test:run`
3. `pnpm test:system-input`（如果本次改动涉及系统输入）
4. `pnpm build`
5. `pnpm tauri build`
6. `pnpm build:desktop`
7. 在目标机器上验证安装、启动、翻译、日志中心、快捷键

发布前需要同步确认这些版本文件：

- [`package.json`](../package.json)
- [`src-tauri/Cargo.toml`](../src-tauri/Cargo.toml)
- `src-tauri/tauri.conf.json`
- [`src-tauri/tauri.smoke.conf.json`](../src-tauri/tauri.smoke.conf.json)（如 Smoke 包对外共享）

## 9. 常见问题

### 9.1 `pnpm tauri dev` 失败

优先检查：

- `1420` 端口是否被占用
- Rust 是否安装完整
- Windows C++ Build Tools 是否安装
- 是否使用了 `Developer PowerShell for VS 2022` 或 `pnpm tauri:dev:windows`
- WebView2 Runtime 是否缺失

典型报错可以这样判断：

- `cargo metadata --no-deps --format-version 1: program not found`
  - 说明 `cargo` 不在当前环境里，先安装 Rust 并确认 `cargo -V`
- `linker 'link.exe' not found`
  - 说明没有安装 `Desktop development with C++`，或当前终端没有加载 `VsDevCmd.bat`

### 9.2 `pnpm tauri build` 失败

建议按这个顺序缩小问题范围：

1. `pnpm build`
2. `pnpm tauri build -- --no-bundle`
3. `pnpm tauri build`

如果是 `MSI` 相关报错，优先回头检查 `VBSCRIPT`。

### 9.3 Smoke 测试找不到可执行文件

优先确认：

- 是否已执行 `pnpm build:desktop:smoke` 或 `pnpm tauri build`
- 可执行文件是否在脚本默认探测路径中
- 是否需要显式传入 `-AppExecutable`

### 9.4 构建能过，但系统输入增强没效果

优先检查：

- 当前是否为 Windows
- 当前构建是否真的包含最新改动
- 配置文件是否写到了预期的数据目录
- 是否命中了白名单 / 黑名单 / 代码编辑器排除

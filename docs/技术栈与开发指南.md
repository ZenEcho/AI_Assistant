# 技术栈与开发指南

这份文档面向后续维护者，目标不是简单列技术名词，而是帮助你快速回答三个问题：

1. 当前项目到底包含哪些运行链路？
2. 一项需求应该改 `page / store / service / Rust` 的哪一层？
3. 这轮新增的自动目标语言、系统输入增强、日志中心分别落在哪些文件里？

## 1. 当前版本的能力版图

相较项目初始版本，当前主线已经不只是“一个带模型设置的翻译窗”，而是一个包含四条核心链路的桌面应用：

- 主翻译链路：文本 / 图片输入 -> 自动语言解析 -> Provider 调用 -> 结果窗展示 -> 历史 / 缓存写入
- 系统输入链路：外部输入框触发 -> 原生捕获 -> 会话编排 -> 回写 / 回退结果窗
- 日志链路：前端 logger -> 脱敏 -> Rust 落盘 -> 实时广播 -> 日志中心查询 / 导出
- 多窗口链路：主窗口、结果窗口、设置窗口、目标语言悬浮窗之间协作

理解这四条链路后，后续改需求基本不会迷路。

## 2. 技术选型与分工

### 前端

- `Vue 3`：页面和组件
- `TypeScript`：类型约束
- `Pinia`：全局状态与跨页面数据
- `Vue Router`：窗口内路由
- `Naive UI`：桌面表单、表格、抽屉等组件
- `UnoCSS`：快速样式组织

### 桌面与原生

- `Tauri 2`：多窗口桌面壳
- `Rust`：Provider 命令、日志持久化、系统语言读取、系统输入增强原生能力
- `tauri-plugin-global-shortcut`：全局快捷键
- `tauri-plugin-store`：本地配置与历史存储
- `tauri-plugin-notification`：系统通知
- `tauri-plugin-autostart`：开机自启动
- `tauri-plugin-single-instance`：单实例

### 存储

- Tauri Store：
  - `app-config.json` 保存模型、偏好、系统输入、日志设置
  - `translation-history.json` 保存翻译历史
- `IndexedDB`：保存翻译缓存
- `<app-data>/logs/*.ndjson`：Rust 侧统一落盘日志
- `BroadcastChannel`：配置与历史的多窗口同步

### 测试与工程化

- `Vitest` + `Vue Test Utils`：前端单元测试
- `cargo test`：Rust 侧原生测试
- `scripts/system-input-smoke-notepad.ps1`：Windows Notepad Smoke 测试
- `scripts/classify-tauri-artifacts.mjs`：打包产物归档

## 3. 运行时窗口模型

项目当前至少涉及四类窗口：

- `main`
  - 主翻译窗口，输入文本 / 图片、选择语言、查看最近记录
- `result`
  - 独立结果窗口，负责流式显示翻译结果，也承担系统输入失败时的回退展示
- `settings`
  - 设置中心，包含模型设置、应用设置、日志中心三大页签
- `target-language-overlay`
  - 系统输入目标语言悬浮窗，通过 `Ctrl+~` 呼出并循环切换目标语言

窗口定义与行为入口主要在：

- [`src/services/window/windowManager.ts`](../src/services/window/windowManager.ts)
- [`src/pages/SettingsWindowPage.vue`](../src/pages/SettingsWindowPage.vue)
- [`src/pages/TranslateResultPage.vue`](../src/pages/TranslateResultPage.vue)
- [`src/pages/SystemInputTargetLanguageOverlayPage.vue`](../src/pages/SystemInputTargetLanguageOverlayPage.vue)

## 4. 目录职责

```text
src/
├─ components/        # 复用组件与应用生命周期控制
├─ constants/         # 默认值、语言选项、日志枚举
├─ pages/             # 页面级组件
├─ router/            # 窗口内路由
├─ services/          # 业务服务层与平台桥接
├─ stores/            # Pinia 全局状态
├─ test/              # 前端单元测试
├─ types/             # 跨层共享类型
└─ utils/             # 主题与通用工具

src-tauri/
├─ capabilities/      # Tauri capability 配置
├─ src/commands/      # 暴露给前端的 Rust 命令
├─ src/logging/       # Rust 日志落盘 / 查询 / 导出
├─ src/system_input/  # 系统输入增强原生实现
├─ src/lib.rs         # Tauri 应用入口与命令注册
└─ tauri*.json        # 标准与 Smoke 打包配置
```

经验法则：

- `pages/` 只做页面拼装与交互绑定
- `stores/` 负责“全局状态 + 持久化同步 + 业务编排”
- `services/` 负责“可复用能力 + Tauri 调用封装 + 平台隔离”
- `src-tauri/` 只处理前端无法稳定完成的原生与桌面能力

## 5. 启动链路

前端入口在 [`src/main.ts`](../src/main.ts)。

当前启动顺序大致如下：

1. 创建 Vue App 与 Pinia。
2. 启动日志桥接 `startLogBridge()`，让 Rust 落盘日志能回流到前端。
3. 初始化 [`src/stores/appConfig.ts`](../src/stores/appConfig.ts)。
4. 将日志运行时配置同步到 Rust。
5. 安装 console / 前端异常捕获。
6. 初始化 [`src/stores/translation.ts`](../src/stores/translation.ts)。
7. 注册 [`src/stores/systemInput.ts`](../src/stores/systemInput.ts)。
8. 应用主题并挂载路由。

应用生命周期附加逻辑在 [`src/components/app/AppLifecycleController.vue`](../src/components/app/AppLifecycleController.vue)，这里负责：

- 系统托盘
- 关闭行为弹窗
- 全局快捷键注册
- 系统输入增强快捷键注册
- 目标语言悬浮窗预热

## 6. 翻译主链路

### 6.1 请求解析

主翻译与系统输入最终都会收敛到 [`src/stores/translation.ts`](../src/stores/translation.ts)。

关键步骤：

1. 调用 `resolveRequest()`
2. 进入 [`src/services/ai/translationRequestResolver.ts`](../src/services/ai/translationRequestResolver.ts)
3. 如果目标语言不是 `auto`，直接生成手动解析结果
4. 如果目标语言是 `auto`：
   - 先取系统语言 [`src/services/app/systemLanguageService.ts`](../src/services/app/systemLanguageService.ts)
   - 再做语言检测 [`src/services/ai/languageDetectionService.ts`](../src/services/ai/languageDetectionService.ts)
   - 最后生成 `resolution`

自动目标语言规则是：

- 源语言与系统语言一致 -> 翻译到 `English`
- 源语言与系统语言不同 -> 翻译到系统语言
- 短文本 / 混合文本 / 低置信度 -> 回退到系统语言

### 6.2 缓存、请求与历史

解析完成后，`translationStore` 会：

1. 先查 [`src/services/storage/translationCacheStorage.ts`](../src/services/storage/translationCacheStorage.ts)
2. 命中缓存则直接返回并写历史
3. 未命中则进入 [`src/services/ai/translationService.ts`](../src/services/ai/translationService.ts)
4. `translationService` 构造 system/user messages
5. 通过 [`src/services/ai/providerFactory.ts`](../src/services/ai/providerFactory.ts) 创建 Provider
6. 当前唯一 Provider 为 [`src/services/ai/providers/openaiCompatibleProvider.ts`](../src/services/ai/providers/openaiCompatibleProvider.ts)
7. Provider 调用 Rust 命令执行流式 / 非流式请求
8. 成功后写入缓存和 [`src/services/storage/translationHistoryStorage.ts`](../src/services/storage/translationHistoryStorage.ts)

### 6.3 图片翻译

图片翻译不是单独的 OCR 服务，而是：

- 前端把图片转成 `dataUrl`
- `translationService` 生成多模态消息
- Provider 以 OpenAI Compatible 的 `image_url` 消息发送
- 自动目标语言与语言检测同样支持图片场景

相关入口：

- [`src/pages/TranslatePage.vue`](../src/pages/TranslatePage.vue)
- [`src/services/ai/translationService.ts`](../src/services/ai/translationService.ts)
- [`src/services/ai/languageDetectionService.ts`](../src/services/ai/languageDetectionService.ts)

## 7. 系统输入增强链路

这是本轮最重的新增能力之一，建议按“前端编排 + Rust 原生”两层理解。

核心文件：

- [`src/stores/systemInput.ts`](../src/stores/systemInput.ts)
- [`src/services/systemInput/nativeBridge.ts`](../src/services/systemInput/nativeBridge.ts)
- [`src/services/systemInput/sessionOrchestrator.ts`](../src/services/systemInput/sessionOrchestrator.ts)

前端职责包括：

- 当前模型优先、默认模型兜底
- 译文复用与失败回退
- 目标语言悬浮窗切换

### 7.2 Rust 原生层

核心文件：

- [`src-tauri/src/commands/system_input.rs`](../src-tauri/src/commands/system_input.rs)

Rust 层负责：

- 原生状态初始化

- 剪贴板读取与粘贴
- 原生替换 / 模拟输入 / 剪贴板回写
- 会话状态与原生状态回传
- 事件发射给前端

当前平台结论：

- Windows：主实现已接入
- macOS / Linux：目录与状态结构已保留，但没有同等完整实现

## 8. 日志链路

日志中心不是一个单独页面，而是一整套基础设施。

主要文件：

- [`src/services/logging/logger.ts`](../src/services/logging/logger.ts)
- [`src/services/logging/logSanitizer.ts`](../src/services/logging/logSanitizer.ts)
- [`src/services/logging/logEmitter.ts`](../src/services/logging/logEmitter.ts)
- [`src/services/logging/logBridge.ts`](../src/services/logging/logBridge.ts)
- [`src/stores/logCenter.ts`](../src/stores/logCenter.ts)
- [`src/pages/SettingsLogCenterPage.vue`](../src/pages/SettingsLogCenterPage.vue)
- [`src-tauri/src/logging/mod.rs`](../src-tauri/src/logging/mod.rs)
- [`src-tauri/src/logging/storage.rs`](../src-tauri/src/logging/storage.rs)

日志主链路：

`page/store/service -> logger -> sanitizer -> app_log_append -> Rust 落盘 -> app-log:created -> logCenterStore -> 设置页日志中心`

日志覆盖面目前包括：

- 翻译请求、缓存命中、Provider 调用
- 设置变更
- 窗口打开/隐藏/聚焦
- 快捷键注册与触发
- 系统输入增强状态与回写
- Vue 运行时错误、`window.onerror`、`unhandledrejection`
- Rust 侧系统输入与网络请求

更多说明见 [logging-system.md](./logging-system.md)。

## 9. 持久化边界

理解“什么该落哪儿”非常重要。

### `app-config.json`

负责保存：

- 主题、语言、关闭行为
- 自启动、全局快捷键、翻译快捷键
- 主翻译与系统输入偏好
- 日志设置
- 模型配置

文件入口：

- [`src/services/storage/appConfigStorage.ts`](../src/services/storage/appConfigStorage.ts)

### `translation-history.json`

负责保存：

- 翻译请求快照
- 解析后的 `resolution`
- 图片来源信息
- 结果文本、模型、Provider、token 使用量

文件入口：

- [`src/services/storage/translationHistoryStorage.ts`](../src/services/storage/translationHistoryStorage.ts)

### IndexedDB 缓存

负责保存：

- 去重后的翻译结果缓存
- 以模型配置 + 请求内容 + 图片哈希为 key
- 按历史上限做裁剪

文件入口：

- [`src/services/storage/translationCacheStorage.ts`](../src/services/storage/translationCacheStorage.ts)

### 日志目录

Rust 统一写入 `<app-data>/logs/*.ndjson`，导出文件写入 `<app-data>/exports/`。

## 10. 常见改动该从哪里下手

### 新增一个应用设置项

通常要改：

- [`src/types/app.ts`](../src/types/app.ts) 或 [`src/types/systemInput.ts`](../src/types/systemInput.ts)
- [`src/constants/app.ts`](../src/constants/app.ts)
- [`src/services/storage/appConfigStorage.ts`](../src/services/storage/appConfigStorage.ts)
- [`src/stores/appConfig.ts`](../src/stores/appConfig.ts)
- 对应设置页面

### 调整自动目标语言规则

优先看：

- [`src/services/ai/translationRequestResolver.ts`](../src/services/ai/translationRequestResolver.ts)
- [`src/services/ai/languageDetectionService.ts`](../src/services/ai/languageDetectionService.ts)
- [`src/services/ai/translationResolutionFormatter.ts`](../src/services/ai/translationResolutionFormatter.ts)
- [`src/constants/languages.ts`](../src/constants/languages.ts)

### 修改系统输入回写策略

优先看：

- [`src/services/systemInput/sessionOrchestrator.ts`](../src/services/systemInput/sessionOrchestrator.ts)
- [`src/services/systemInput/strategy.ts`](../src/services/systemInput/strategy.ts)
- [`src-tauri/src/system_input/platform/windows.rs`](../src-tauri/src/system_input/platform/windows.rs)

### 新增日志点

优先做法：

1. 在合适的 `page / store / service / rust command` 中创建 logger
2. 补充 `category`、`source`、`requestId / traceId`
3. 确认敏感信息经过 `logSanitizer`
4. 如有必要，再更新日志中心文档

### 新增一个原生命令

建议顺序：

1. 在 `src-tauri/src/commands/*` 暴露 `#[tauri::command]`
2. 在 [`src-tauri/src/lib.rs`](../src-tauri/src/lib.rs) 注册
3. 在 `src/services/*` 新增前端 bridge
4. 页面只调用 service，不直接散落 `invoke`

## 11. 当前边界与注意事项

- 系统输入增强当前应按 Windows 主线能力维护，跨平台不要轻易在 README 中承诺“等价支持”。
- 自动目标语言依赖系统语言与语言检测结果，文案和行为必须同时更新，不能只改其中一层。
- 日志系统默认已经具备排障价值，新增埋点时优先保证结构化与脱敏，而不是堆原始文本。
- `TranslatePage` 与系统输入都会触发翻译，但两条链路不能相互污染状态，改动时要留意 `translate()` 与 `translateDetached()` 的差异。
